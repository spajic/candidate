  Единая система газоснабжения (ЕСГ) России представляет собой

  % 2. АСДУ управляет ЕСГ
  Иерархическое управление ЕСГ осуществляется 
  автоматизированной системой диспетчерского управления (АСДУ), состоящей
  из трёх уровней:
  \begin{enumerate}
    \item
      ЦПДД – Центральный производственно-диспетчерский департамент ОАО «Газпром»;
    \item 
      ПДС ЭО – производственно-диспетчерская служба эксплуатирующей организации;
    \item
      ДП ЛПУ МГ – диспетчерский пункт линейного производственного управления 
    магистральными газопроводами.  
  \end{enumerate}

  Важную роль в АСДУ играет система поддержки принятия диспетчерских решений (СППДР),
  состоящая из многочисленных программно-вычислительных комплексов (ПВК) СППДР,

  \begin{itemize}

    \item

  Для реализации перечисленных выше требований при разработке архитектуры 
  вычислительной среды были приняты три ключевых принципа.
  \begin{enumerate}
    \item %l: Сервис-ориентированный подход средствами middleware
      Реализация сервис-ориентированного подхода
      на основе использования компонентного программного обеспечения 
      промежуточного уровня (middleware).
    \item %l: Поддержка явного управления ресурсами (термин раскрывается ниже)
      Обеспечение явного управления вычислительными ресурсами.
    \item %l: Универсальный алгоритм разработки сервисов на базе среды.
      Разработка вычислительных сервисов для среды 
      осуществляется по унифицированному алгоритму, 
      позволяющему максимально использовать её преимущества.
  \end{enumerate}

%l: Ввод используемых терминов ------------------------------------------------
  %l: УЗЕЛ - компьютер, входящий в вычислительную среду
    Основными ресурсами распределённой вычислительной среды являются компьютеры, 
    связанные сетями передачи данных и называемые узлами.

    Каждый узел распределённой системы:
    \begin{itemize}
    \item %l: Вычислительные ресурсы
      содержит набор вычислительных ресурсов: центральных процессоров (ЦП); 
      GPGPU-видеокарт (General Purpose Graphics Processing Unit); 
      других устройств, способных осуществлять вычисления;
    \item %l: ОС
    работает под управлением той или иной операционной системы: 
    Windows, Linux, Mac OS и так далее;
    \item %l: ЯП, ПО, тезнологии параллельного программирования
    выполняет ряд приложений, написанных с использованием тех или иных 
    языков программирования (C++, Java, C\#, Objective-С,…) и 
    технологий параллельного программирования (OpenMP, MPI, TBB, CUDA, …);
    \item %l: Middleware
    содержит <<Слой удалённого взаимодействия>>~--- 
    специализированное программное обеспечение, выступающее посредником между 
    прикладными приложениями и низкоуровневыми функциями сетевого взаимодействия.
    \end{itemize}

  %l: СЕРВИС - это 
     <<Сервис>> - это программный компонент, 
     состоящий из двух частей: интерфейса и исполнителя. 
     Интерфейс сервиса представляет собой описание его методов 
     на специализированном языке описания интерфейсов. 
     Исполнитель сервиса представляет собой программную реализацию 
     заявленных в интерфейсе методов, функционирующую в рамках 
     вычислительных ресурсов узла распределённой системы для того, 
     чтобы принимать и обрабатывать удалённые вызовы методов сервиса.

  %l: Взаимоотношение клиентов и серверов - компонент может играть и ту и ту роль
    Компоненты распределённой системы могут выступать 
    в роли клиента и в роли сервера. 
    Сервер~--– это компонент, предоставляющий некоторые сервисы, 
    а клиент~--– компонент, использующий предоставленные сервисы. 
    Один и тот же компонент может одновременно выступать 
    как в роли клиента, так и в роли сервера: 
      предоставлять некоторые методы для клиента, 
      и использовать для их выполнения другие сервисы.

  %l: низкоуровневое взаимодействие- middleware   
    Низкоуровневые механизмы взаимодействия клиента и сервера 
    выполняются слоем удалённого взаимодействия.

  %l: Преимущества SOA: прозрачность расположения, реализации, платформы
    Организация системы в виде сервисов, взаимодействующих в распределённой среде, 
    имеет следующие преимущества: 
    прозрачность 
     расположения, 
     реализации и 
     платформы 
    компонент распределённой системы.

\emph{Прозрачность расположения компонент.} 
  Сервисы изначально ориентированы на удалённое взаимодействие. 
  Они взаимодействуют по одинаковой схеме 
  и при работе в рамках единого узла, и на разных узлах.

\emph{Прозрачность реализации компонент.} 
  Функциональность сервиса полностью определяется его интерфейсом. 
  При этом она может быть реализована с использованием различных 
  технологий, алгоритмов, выполняться на различных вычислительных ресурсах.

\emph{Прозрачность платформы компонент.} 
  Компоненты распределённой системы могут работать и взаимодействовать 
  под управлением различных операционных систем, 
  могут быть созданы с использованием различных языков программирования. 
  Такая гибкость достигается благодаря слою удалённого взаимодействия, 
  выполняющего роль адаптера между клиентом и сервером.

% Явное управление вычислительными ресурсами ----------------------------------
\paragraph{Явное управление вычислительными ресурсами.}

%l: Сервисы подразделены на СЛУЖЕБНЫЕ и ВЫЧИСЛИТЕЛЬНЫЕ
  Сервисы в системе подразделяются на \emph{служебные} и \emph{вычислительные}. 

%l: ВЫЧИСЛИТЕЛЬНЫЕ СЕРВИСЫ - выполняют вычисления
  \emph{Вычислительные сервисы} 
  осуществляют вычисления, направленные на решение расчётных задач.

%l: СЛУЖЕБНЫЕ СЕРВИСЫ - организуют взаимодействие клиентов, сервисов, КВР
  \emph{Служебные сервисы} 
  организуют взаимодействие клиентов, сервисов и вычислительных ресурсов 
  в распределённой среде.

%l: КВР - совокупность выч. рес. М.б. из нескольких узлов/подмножество одного
  \emph{Комплект вычислительных ресурсов}~--– 
  это совокупность вычислительных ресурсов: 
  возможно, подмножество вычислительных ресурсов одного узла~--– 
  возможно, объединение вычислительных ресурсов нескольких узлов.

%l: ТИП КВР - хар-ка КВР, позволяющая сопоставить КВР и мн-во выч. сервисов
  \emph{Тип комплекта вычислительных ресурсов} 
  присваивается каждому комплекту вычислительных ресурсов с целью сопоставить 
  множества вычислительных сервисов и комплектов вычислительных ресурсов: 
  для каждого вычислительного сервиса 
  задаётся перечень типов комплектов ресурсов, 
  приспособленных для его эффективного выполнения.

%l: СЛУЖЕБНЫЙ СЕРВИС УПРАВЛЕНИЯ ТИПИЗИРОВАННЫМ КВР - 
  \emph{Служебный сервис 
  управления типизированным комплектом вычислительных ресурсов} 
  выполняется в рамках каждого комплекта вычислительных ресурсов и 
  предоставляет методы для запуска вычислительных сервисов 
  в рамках управляемого комплекта вычислительных ресурсов. 
  Множество доступных вычислительных сервисов определяется 
  типом комплекта вычислительных ресурсов.

%l: СЛУЖЕБНЫЙ СЕРВИС ДИСПЕТЧЕР РЕСУРСОВ - 
  \emph{Служебный сервис диспетчер ресурсов.} 
  Выполняется на одном из узлов распределённой системы и 
  управляет комплектами типизированных вычислительных ресурсов 
  в рамках множества управляемых узлов. 
  Его интерфейс предоставляет возможность 
  монопольного захвата комплекта вычислительных ресурсов необходимого типа.

%l: Алгоритм работы схемы
  Предложенная схема функционирует по следующему алгоритму.
  %

%l: Этап настройки - установка, формирование КВР, запуск УС
  На этапе настройки среды на узлах, входящих в распределённую систему, 
  формируются типизированные комплекты вычислительных ресурсов и запускаются 
  соответствующие служебные сервисы управления комплектами ресурсов.
%l: Запуск диспетчера, определения мн-ва узлов, управляемых им.
  На одном из узлов запускается диспетчер ресурсов. 
  Определяется множество узлов, которыми он управляет. 
  Все управляющие сервисы комплектов  вычислительных ресурсов собираются 
  сервис-диспетчером в типизированные пулы.

%l: Действия клиента для решения задачи.
  Для осуществления своих вычислительных потребностей клиент 
  инициирует следующую последовательность действий:
  \begin{enumerate}
  \item %l: Запрос КВР необходимого типа у диспетчера
    Клиент запрашивает у сервис-диспетчера 
    комплект вычислительных ресурсов необходимого типа.
  \item %l: Установление прямого монопольного канала - Захват
    Сервис-диспетчер обеспечивает установление прямого соединения 
    между клиентом и управляющим сервисом соответствующего типа и 
    монопольный характер захвата.
  \item %l: Вызов методов УС для запуска вычислительных сервисов.
    Клиент использует методы сервиса, 
    управляющего захваченным комплектом вычислительных ресурсов, 
    для запуска необходимых вычислительных сервисов.
  \item %l: Вызов методов запущенных ВС для выполнения работы.
    Клиент использует запущенные вычислительные сервисы 
    для выполнения необходимой работы.
  \item %l: Освобождение захваченного ресурса и возвращение его в соотв. пул
    Клиент освобождает захваченный вычислительный ресурс и 
    его управляющий сервис возвращается сервис-диспетчером в соответствующий пул.
  \end{enumerate}

%l: Что даёт явное управление ВР: HPC; мин. расходы; конф. в распр. среде
  Явное управление вычислительными ресурсами позволяет обеспечить 
    поддержку высокопроизводительных вычислений, 
    минимизировать накладные расходы удалённого взаимодействия сервисов, 
    управлять организацией вычислительного процесса в распределённой среде. 

%l: HPC: Сопоставть HPC-сервисы соотв. КВР и технологиям
 \emph{Высокопроизводительные вычисления} 
  требуют сопоставления вычислительным сервисам, 
  использующим технологии параллельного и распределённого программирования, 
  соответствующих вычислительных ресурсов. 
  Оно обеспечивается за счёт явного сопоставления 
  вычислительных сервисов и комплектов вычислительных ресурсов на основе типов 
  таким образом, чтобы вычислительные сервисы выполнялись на оборудовании, 
  поддерживающем необходимые технологии параллельных вычислений.

%l: Мин. расходы: монопольн. + прямой + сохранение состояния.
 \emph{Минимизация накладных расходов удалённого взаимодействия} 
  достигается посредством обеспечения возможности 
    монопольного захвата вычислительных ресурсов, 
    прямой работы клиента с сервером, 
    операций с сохранением состояния.

%l: Монопольн: HPC -> один полностью исп-ет. CUDA - must.
  Монопольная работа с вычислительным ресурсом целесообразна 
  при ориентации на высокопроизводительные вычисления, так как 
  уже один клиент может полностью использовать потенциал 
  вычислительного ресурса. 
  При работе с CUDA-ресурсами обеспечение монопольного доступа необходимо 
  по причине недостаточных механизмов прерываний, защиты памяти и так далее.

%l: Statefull из монопольн. Пример.
  Монопольный характер доступа наряду с 
  прямым доступом клиента к захваченному ресурсу позволяет 
  с минимальными накладными расходами 
  осуществлять операции с сохранением состояния. 
  Состояние исполнителя сервиса может изменяться только клиентом, 
  осуществившим его захват~--– благодаря этому клиент может осуществлять 
  последовательность действий над удалённым объектом 
  при гарантированном сохранении его состояния. 
  Например, когда требуется последовательно рассчитать 
  на одном вычислительном ресурсе несколько режимов СГ, 
  отличающихся только краевыми значениями параметров газовых потоков, 
  клиент может просто изменять значение краевых параметров газовых потоков и 
  проводить новый расчёт режима СГ. 
  Без этой возможности необходимо было бы каждый раз передавать 
  сервису расчёта режима целиком все множество параметров объектов СГ 
  с новыми краевыми параметрами газовых потоков, 
  что влекло бы за собой огромные накладные расходы.

%l: Контроль над организацией выч. процесса за счёт выбора клиентом диспетчера
  \emph{Контроль над 
  организацией вычислительного процесса в распределённой среде} 
  достигается за счёт 
  выбора клиентом необходимого диспетчера вычислительных ресурсов, 
  организующего соответствующий режим работы:

%l: Различные режимы работы за счёт расположения диспетчера и клиента
  \begin{itemize}
    \item \emph{однопользовательский режим}~--- 
      диспетчер и управляемые им комплекты вычислительных ресурсов 
      находятся на компьютере клиента; 
      набор доступных вычислительных сервисов ограничивается типом комплекта 
      вычислительных ресурсов клиента;

    \item \emph{клиент-серверный режим}~--- 
      вычислительные ресурсы выделенного сервера моделирования 
      (множество узлов, обладающих вычислительными ресурсами, достаточными для 
      одновременного обслуживания нескольких клиентов) 
      управляются одним сервис-диспетчером. 
      Клиенты освобождены от вычислительной нагрузки и 
      запрашивают вычислительные ресурсы у сервис-диспетчера. 
      В качестве сервера может использоваться «облачный» вычислительный сервер
      с «эластичными» вычислительными ресурсами.

    \item 
    \emph{использование вычислительных ресурсов клиентов в локальной сети}~---
      на компьютерах клиентов формируются комплекты вычислительных ресурсов; 
      на одном из компьютеров запускается сервис-диспетчер, 
      управляющий всеми ресурсами.
      Сервис-диспетчер стремится выделить каждому клиенту его собственный 
      комплект ресурсов. 
      Каждый клиент получает возможность расширить набор доступных ему 
      вычислительных ресурсов за счёт свободных мощностей соседних компьютеров.

    \item \emph{распределённый режим} – 
      возможна организация вычислительного процесса в распределённой системе, 
      соответствующей иерархии диспетчерского управления. 
      Комплекты вычислительных ресурсов, 
      расположенные на серверах соответствующих уровням диспетчерских служб~--- 
      ЦПДД, ПДС ЭО, ДП ЛПУ~--- управляются соответствующими сервис-диспетчерами. 
      Если клиенту, работающему на серверах ПДС ЭО, требуется 
      для решения вычислительно-сложной задачи 
      мощный комплект вычислительных ресурсов, располагаемый в ЦПДД~--– 
      он запрашивает его у сервис-диспетчера уровня ЦПДД. 
      Если клиенту на уровне ПДС ЭО требуется организовать вычислительный процесс, 
      отдельные компоненты которого будут выполняться на вычислительных ресурсах 
      нескольких ДП ЛПУ~--– 
      он обращается к соответствующим сервис-диспетчерам, 
      захватывает требуемые вычислительные ресурсы на серверах ДП ЛПУ, 
      запускает и использует на них необходимые вычислительные сервисы.

    \item \emph{комбинированный режим} – 
      перечисленные выше режимы можно комбинировать, например, 
      при организации иерархического процесса взаимодействия 
      ПДС~ЭО~–-- ДП~ЛПУ на уровне ДП~ЛПУ расчетные задачи могут решаться 
      с использованием локальных вычислительных ресурсов.
  \end{itemize}

%l: Алгоритм разработки вычислительных сервисов в разработанной среде----------
  \section*{Алгоритм разработки вычислительных сервисов 
  в предложенной среде моделирования}

%l: Какие задачи стоят перед алгоритмом
  Алгоритм разработки вычислительных сервисов в предложенной среде 
  моделирования направлен на достижение следующих качеств:
  \begin{itemize}
  \item %l: сервисы соответствуют архитектуре, исп-ют её преимущества
  обеспечение соответствия вновь разрабатываемых сервисов идеологии 
  вычислительной среды и максимальное использование её преимуществ;
  \item %l: сохранение лёгкости внесения изменений по мере внедрения новых С-в
  сохранение лёгкости и простоты внесения изменений в систему по мере 
  разработки новых вычислительных сервисов;
  \item %l: высокая производительность
  обеспечение высокой производительности разрабатываемых сервисов.
  \end{itemize}

%l: Этапы алгритма:
 Алгоритм состоит из следующих этапов.

 \emph{Проектирование схемы классов} 
   решения задачи осуществляется на языке моделирования UML 
   с использованием принципов и шаблонов 
   объектно-ориентированного программирования 
   на трёх взаимосвязанных уровнях: концепций, интерфейсов и реализаций.

 \emph{На уровне концепций} 
    выявляются основные взаимодействующие сущности, 
    их ответственности и взаимодействия для решения поставленной задачи. 
    На основе анализа вариаций основных концепций выделяются части программы, 
    подверженные изменениям~--– они инкапсулируются 
    в виде семейств взаимозаменяемых классов.

\emph{На уровне интерфейсов} 
  разрабатываются абстрактные интерфейсы, 
  описывающие взаимодействие выявленных концепций для решения задачи. 
  Результатом этого уровня становится набор абстрактных интерфейсных классов.

\emph{На этапе реализации} 
  уточняются требования к интерфейсам, вносятся соответствующие изменения.
  %

\emph{Разработка тестов в терминах абстрактных интерфейсов}
  для выполнения в системе автоматического тестирования 
  должна осуществляться до реализации,
  что позволяет тестировать реализации интерфейсов единым образом.

\emph{Реализация абстрактных интерфейсов вычислительными сервисами }
  осуществляется на следующих этапах:
  \begin{itemize}
    \item 
      разрабатывается интерфейс вычислительного сервиса, совместимый 
      с разработанным абстрактным интерфейсом решения расчетной задачи;
    \item 
      разрабатывается исполнитель сервиса, реализующий интерфейсные методы;
    \item 
      определяются типы вычислительных ресурсов, 
      подходящих для выполнения разработанного сервиса; 
      в интерфейсы соответствующих сервисов управления ресурсами добавляются 
      методы запуска разработанного сервиса;
    \item 
      разрабатывается клиентский класс для сервиса, 
      реализующий абстрактный интерфейс, 
      и выполняющий его методы за счёт установления связи с 
      сервисом-исполнителем и переадресации вызовов методов ему.
  \end{itemize}

\emph{Тестирование корректности и производительности} 
  разработанного сервиса осуществляется на основе разработанных сервисов 
  в автоматической системе тестирования. 
  Реализация сервиса дорабатывается до тех пор, пока она не сможет 
  удовлетворить всем сформулированным тестам.

\emph{Внесение изменений в систему} 
  может потребоваться в процессе эксплуатации.
  Изменения, вносимые в систему, можно разделить на две группы: 
  без изменения функциональности~--- 
    улучшение архитектуры классов, 
    скорости и эффективности выполнения задач; 
  c изменением функциональности~--- 
    новая функциональность, 
    модификация существующей, 
    устранение ошибок.

При работе без изменения функциональности целостность системы 
контролируется работой автоматических тестов: 
ни один тест не должен перестать показывать корректность работы. 
При необходимости изменения функциональности системы 
происходит возвращение на этап проектирования. 

В результате исполнения предложенной схемы система строится из компонент, 
отвечающих требованиям 
  высокой связности внутри компонент и 
  низкой степени взаимозависимости между компонентами. 
Лёгкость внесения изменений, гибкость системы 
обеспечивается соблюдением принципов и применением шаблонов 
объектно-ориентированного проектирования. 

Решение отдельной вычислительной задачи, 
в отличие от монолитной архитектуры ПВК, 
представляется в виде совокупности сервисов, 
которые динамически запускаются и останавливаются в распределённой среде.

\section*{Проектирование вычислительных процессов 
 решения задач моделирования режимов систем газоснабжения}

%l: Базовый комплекс задач
  В работе проведён анализ структуры задач моделирования режимов СГ 
  и установлено, что может быть выделен базовый комплекс расчетных задач, 
  включающий в себя следующие основные процедуры: 
  \begin{itemize}
    \item 
      расчёт свойств газа и газового потока;
    \item 
      процедуры моделирования режимов отдельных объектов СГ: 
        трубопроводы, 
        крановые системы, 
        аппараты воздушного охлаждения, 
        газоперекачивающие агрегаты;
    \item 
      трубопроводные системы, 
      компрессорные цеха, 
      магистральные газопроводы, 
      газотранспортные системы.
  \end{itemize}

%l: Задачи более высокого уровня основаны на базовом комплексе 
  На основе процедур базового комплекса расчетных задач решаются 
  задачи более высокого уровня, в частности 
    задачи оптимизации и прогнозирования режимов СГ, 
    мониторинга фактического состояния СГ и 
    ситуационного анализа.

%l: Какие задачи спроектированы и реализованы в работе:
  В диссертационной работе спроектированы 
  распараллеленные, распределенные вычислительные процессы решения задачи 
    моделирования режимов СГ~и 
  адаптации модели режимов СГ к фактическим режимам.

% Сервис моделирования режимов СГ ---------------------------------------------
\subsection*{
Вычислительный процесс моделирования режимов системы газоснабжения}

%l: Постановка задач моделирования режимов СГ
  Для решения различных расчётных задач моделирования режимов 
  любая система газоснабжения представляется 
  условно-ориентированным «расчётным графом» 
    (дуги имеют направление потоков газа, 
    но оно может изменяться в процессе расчётов). 
  Дугам графа соответствуют моделируемые технологические объекты, 
  участвующие в перекачке газа (
    трубопроводы, 
    крановые системы, 
    ГПА, 
    КЦ
    и так далее),~--– 
  узлы графа соединяют входы/выходы дуг~--- 
  соответствующих моделируемых объектов.

%l: Общий вид процедуры моделирования объекта
  Для каждого моделируемого объекта создана расчётная процедура вида 
  \eqref{eq:_common_object_model}:
  \begin{equation}\label{eq:_common_object_model} % Общий вид моделир-го объекта
    \{  
        T_{ \text{вых} }, Q  
    \} = 
    f 
      \left(
        P_\text{ вх  }, 
        P_\text{ вых }, 
        T_\text{ вх  }, 
        Gas, 
        NSI
      \right)
  \end{equation}

%l: Расшифровка формулы 
  Процедура~\eqref{eq:_common_object_model} позволяет рассчитать 
  расход и температуру газа на выходе объекта 
  по заданным параметрам: 
    давления на входе и выходе, 
    температура на входе, 
    компонентный состав газового потока, 
    параметры внешней среды и 
    параметры объекта. 
  Параметры  объекта включают:  
    нормативно-справочные (паспортные) данные, 
    параметры фактического состояния объекта и 
    режимно-технологические ограничения. 

%l: Задача расчёта режима СГ
\subsection*{Задача расчёта режима системы газоснабжения}

%l: Постановка задачи - что дано
  Задан расчётный граф: 
    топология расчётных объектов, 
    направления потоков. 
  В краевых узлах графа заданы параметры газового потока, 
  необходимые для моделирования режима.

%l: Постановка задачи - что требуется найти
  Требуется определить поле давлений и температур во всех узлах графа,
   обеспечивающее баланс потоков по дугам во всех узлах графа 
   и соблюдение всех режимно-технологических ограничений расчетных объектов.

%l: Метод решения задачи - метод узловых давлений
Эта задача решается методом узловых давлений, состоящим из следующих этапов:
%l: Этапы метода узловых давлений
  \begin{enumerate}
    \item % 1. Начальное приближение температур и давлений
      Задание начального поля давлений и температуры газа 
      $P_k^0, T_k^0$ 
      в узлах графа.
    \item % 2. Расчёт всех расходов
      Расчет по каждой дуге расхода потока газа 
      по давлениям газа в краевых узлах дуги 
      $
        q_i^l (
          P_{ i,\text{вх}  }^l, 
          P_{ i,\text{вых} }^l
        )
      $, 
      $i$ - номер дуги, $l$ - номер итерации.

    \item % 3. Проверка условия завершения
      Мониторинг выполнения условия максимально-допустимого дисбаланса 
      потоков газа в узлах расчетного графа СГ.
    \begin{equation}\label{eq:condition_of_balance_in_every_node}   
      \big| \delta q_{k}^{ l+1 } \big| < \varepsilon_{q}, 
      \text{k - номер узла.}
    \end{equation}

    \item % 4. Формирование СЛАУ
      Если условие \eqref{eq:condition_of_balance_in_every_node} 
      для какого-либо узла не выполняется, 
      то формируем систему уравнений баланса потоков газа в каждом k-м узле 
      графа в соответствии с 1-м законом Кирхгофа:
      \begin{equation}\label{eq:Kirhgof_Law}
        \delta q_{k}^{ l+1 } =
          \sum_{i=1}^{ N_{k} }
            q_{i,k}^{l+1}
            \left(
                P_{ i,{\text{вх}}  }^{ l+1 }, \,
                P_{ i,{\text{вых}} }^{ l+1 }
            \right)-Q_{k}=0,
      \end{equation}
    где $Q_k$~--- заданные притоки или отборы газа в $k$-м узле.

    Новое приближение значения давления 
    $P_k^{l+1}$ в $k$-м узле на $l+1$-й итерации 
    можно представить как 
    $P_k^{l+1}=P_k^{l} + g \cdot \Delta P_k^{l+1}$ 
    ($g$ - регуляризационный параметр). 

    Запишем линейную компоненту разложения 
    $q_i^{l+1}(P_{i,\text{вх}}^l, P_{i,\text{вых}}^l)$ в ряд Тейлора:

    \begin{equation}\label{eq:q_Taylor_series}
      q_i^{l+1}
      \left(
        P_{ i,\text{вх}  }^l, 
        P_{ i,\text{вых} }^l
      \right) \simeq
      q_i^l + 
      \frac
        { 
          \partial q_i^l
        }
        {
          \partial P_{ i, \text{вх} }^{l+1}
        } 
      \Delta P_{ i,\text{вх} }^{l+1} +
      \frac
        {
          \partial q_i^l
        }
        {
          \partial P_{ i, \text{вых} }^{l+1}
        } 
      \Delta P_{ i,\text{вых} }^{l+1}    
    \end{equation}
    и подставим в соответствующие уравнения \eqref{eq:Kirhgof_Law}. 
    Получим систему линейных уравнений относительно $\Delta P_k^{l+1}$.

    \item % 5. Решение СЛАУ
      Решение системы линейных уравнений и расчёт 
      нового поля давлений и температуры 
      $P_k^{l+1}$, $T_k^{l+1}$ 
      газа в узлах графа
      \begin{equation}
        \label{eq:SLAE_of_balances_next_iter}
        P_k^{l+1} = P_k^l + g \cdot \Delta P_k^{l+1}
      \end{equation}

    \item % 6. Расчёт расходов по всем дугам при новых давлениях  
      Расчет по каждой дуге 
      расхода потока газа по давлениям газа в краевых узлах дуги 
      $q_i^{l+1} 
        (
          P_{i,\text{вх} }^{l+1}, P_{i,\text{вых} }^{l+1}
        ), 
      i$ – номер дуги. 
      Если для каких-то объектов 
      происходит нарушение режимно-технологических ограничений, 
      то корректируем значения 
      \eqref{eq:SLAE_of_balances_next_iter} параметром $g$. 
      Вычисляем расчетный суммарный дисбаланс потоков газа в узлах графа
      \begin{equation}\label{eq:disbalace_sum_on_next_iter}
        F^{l+1} = \sum_{k=1}^{M} \delta q_k^{k+1}
      \end{equation}

      Если $F^{l+1}>F^l$, подбираем регуляризационный коэффициент $g$, 
      чтобы условие выполнилось, 
      и переходим к шагу 3.
    \end{enumerate}

%l: Выч. сервис адаптации модели (Идентификации пар-в модели) -----------------
\subsection*{
  Задача адаптации модели системы газоснабжения к фактическим режимам}

%l: Что такое задача адаптации (идентификации)
  Адаптация моделей режима СГ проводится с целью повышения адекватности 
  расчётных результатов модели. 
  Результатом адаптации становится приближение расчётных параметров режима СГ 
  к замеряемым значениям параметров газового потока в системе. 
%l: Почему она выбрана для реализации - вычислительно сложная, м.б. ускорена  
  Решение этой задачи требует большого объёма вычислений 
  и может быть многократно ускорено с использованием 
  параллельных и распределённых вычислений 
  в разработанной автором среде моделирования.

%l: Входные данные для задачи - моделиров-е СГ + замеры + эмпир-е пар-ы
  Для решения задачи адаптации модели должны быть заданы все те же данные, 
  что и для расчёта режима СГ, а также дополнительная, 
  избыточная информация~--–
  множество значений замеров параметров газового потока в некоторых 
  контрольных узлах СГ. 
%l: В качестве замерные значения часто задаются
  В качестве замеров могут быть заданы следующие значения:
  \begin{itemize}
    \item 
      $P_i^{*}, T_i^{*}$ -
      замеры давления и температуры газа на входах и выходах КЦ, ГПА;
    \item $Q_i^{*}$ - 
      расходы газового потока, замеряемые на газо-измерительных станциях (ГИС).
  \end{itemize}

%l: В качестве идентифицируемых эмпирических параметров исп-ся:
  В качестве идентифицируемых параметров модели часто используются:  
  \begin{itemize}
    \item  
      $K_{\text{то}}, K_{\text{эф}}$ - 
      коэффициенты теплообмена и гидравлической эффективности трубопроводов;
    \item  $K_{\eta}, K_N$ - 
      интегральные по КЦ поправочные коэффициенты 
      на газодинамические характеристики 
      центробежных нагнетателей (ЦБН) однотипных ГПА, 
      индивидуальные коэффициенты технического состояния ГТУ и ЦБН ГПА 
      и так далее.
  \end{itemize}

%l: Ввод необходимых обозначений: идентиф-е, расчётные, замеряемые пар-ры
  Обозначим вектор идентифицируемых параметров расчетной модели как 
  $\vec{a} \in  \mathbb{R}^{n_a}, 
  {n_a}$ – количество таких параметров; 
  обозначим векторы замеряемых и расчётных параметров адаптации режима 
  соответственно
    $\vec{p}^{*} \in \mathbb{R}^{n_p}$ и 
    $\vec{p} \in \mathbb{R}^{n_p}$,
    где $n_p$ - количество сравниваемых параметров $\vec{p}^{*}$ и $\vec{p}$.

%l: В кач-ве пар-в адаптации часто используют (повтор)
  В качестве параметров адаптации режима обычно выбирают: 
  расчетные давления, 
  температуру~и 
  расход газа в узлах графа, 
  для которых имеются соответствующие замеры фактических значений.

%l: p=f(a)
  Множество расчётных параметров $\vec{p}$ модели процесса 
  является функцией идентифицируемых параметров $\vec{a}$:
  \begin{equation}\label{eq:adaptation_model_of_GTS}
    \vec{p} = 
      f
      \left(
        \vec{a}
      \right)
  \end{equation}

%l: Минимизация критерия рассогласования p* и p, МНК ()
%ToDo - используется МНК в качестве критерия - плохо сказано
  Задача адаптации модели в общем виде сводится к минимизации некоторого 
  критерия рассогласования 
  $\vec{p}^{*}$ и $\vec{p}$, 
  в качестве которого обычно используют метод наименьших квадратов
  \begin{equation}\label{eq:adaptation_least_squares}
    F = \min_{\vec{a}} \sum_{r=1,\dots n_p} 
      \left( 
        \frac { p_r^{*} - p_r( \vec{a} ) } 
              { p_r^{*} }
      \right)^2 
  \end{equation}

%l: Решение задачи минимизации рассогласов-я можно методом чувствительности
  Решать задачу~\eqref{eq:adaptation_least_squares} можно различными методами,
  однако наиболее подходящим для распараллеливания вычислительного процесса 
  является метод чувствительности, 
  который основан на использовании 
  линейной компоненты численного разложения 
  функционала~\eqref{eq:adaptation_model_of_GTS} в ряд Тейлора:

  \begin{equation}\label{eq:adatptation_Taylor_series}
    p_r(\vec{a}^{l+1}) = 
      p_r(\vec{a}^l) + 
        \sum_{i=1}^{n_a}
          \left( \frac
            {\partial p_r(a^l)} 
            {\partial a_i^l} 
          \Delta a_i^{l+1}
          \right) 
  \end{equation}

%l: Выкладки для сведения к СЛАУ: ряд Тейлора в критерий НК
  Подставляя \eqref{eq:adatptation_Taylor_series} в 
  \eqref{eq:adaptation_least_squares} получаем:

  \begin{equation}\label{eq:adaptation_F_to_minimize}
    F = \min_{\vec a}
    \sum_{r=1\dots n_p}
    \left(
      \frac 
        { \left(
            p_r^* - p_r(\vec{a}^l) - 
              \sum_{i=1}^{n_a} 
                \Big[ \frac{\partial p_r(a^l)} 
                       {\partial a_i^l} 
                \Delta a_i^{l+1}\Big]
          \right)^2 
        }
        { {p_r^{*}}^2 }
    \right)
  \end{equation}  

%l: Условие минимума функционала - СЛАУ производных = 0
  Условием минимума F являестся система уравнений
  \begin{equation}\label{eq:adaptation_min_condition_SLAE}
    \frac { \partial F^{l+1} }
          { \partial \Delta a_k^{l+1} }
     \text{ для } k = 1, \dots n_a
  \end{equation}

%l: Ещё выкладки
  Дифференцируем \eqref{eq:adaptation_F_to_minimize} и запишем 
  \eqref{eq:adaptation_min_condition_SLAE} в следующем виде:
  \begin{equation}\label{eq:adaptation_chuvst_prev}
    \sum_{r=1}^{n_p}   
       \frac{1} {p_r^{*}}
       \left(
          p_r^{*} - p_r^l - 
            \sum_{i=1}^{n_a} 
              \left(\frac {\partial p_r^l} {\partial a_i^l} \Delta a_i^{l+1} \right)
          \left(
            - \frac{\partial p_r^l} {\partial a_k^l}
          \right)
       \right) = 0, k = 1 \dots n_a
  \end{equation}

%l: Ещё выкладки и получили результат
  Перенесём в \eqref{eq:adaptation_chuvst_prev} коэффициенты при 
  $\Delta a_i^{l+1}$ в левую часть и получим:
  \begin{equation}\label{eq:adaptation_chuvst_final}
    \sum_{i=1}^{n_a}  
      \sum_{r=1}^{n_p} 
      \left(
        \frac{1} {p_r^{*}} 
         \frac{\partial p_r^l} {\partial a_i^l} 
         \frac{\partial p_r^l} {\partial a_k^l} 
      \right)  
      =
    \sum_{r=1}^{n_p}   
       \frac{1} {p_r^{*}}     
       \left( p_r^{*} - p_r^l \right)          
       \left( \frac{\partial p_r^l} {\partial a_k^l} \right) , k = 1 \dots n_a
  \end{equation}

%l: Частные производные аппроксимируем разностным отношением
  Частные производные  
  $\frac {\partial p_r^l} {\partial a_k}$ в \eqref{eq:adaptation_chuvst_final}
  можно численно оценить, используя разностные операторы:
  \begin{equation}\label{eq:partial_derivative_numerical}
    \frac {\partial p_r} {\partial a_k} = 
    \frac 
      { 
        f
          \left( 
              \vec{a}_k^{+} 
          \right) 
        - 
        f
          \left( 
              \vec{a}_k 
          \right) 
      } 
      {\delta},
  \end{equation}
  где 
  $\vec{a}_k^{+} = \left(a_1, a_2, \dots, a_k + \delta, \dots, a_{n_a} \right)$;
  $\vec{a}_k     = \left(a_1, a_2, \dots, a_k +       , \dots, a_{n_a} \right)$.

%l: Можно применить центральную разность - точнее, сложнее, больше эффект
  Можно применить разностные операторы более высокого порядка точности, 
  однако это потребует дополнительных вычислительных ресурсов, 
  хотя при этом эффект распараллеливания процесса будет выше.

%l: Итарационная процедура построения решения
  На основе итерационной процедуры 
  формирования системы линейных уравнений 
  \eqref{eq:adaptation_chuvst_final}, 
  вычислении поправок $\Delta a_i^{l+1}$ 
  к идентифицируемым параметрам модели 
  \begin{equation}\label{eq:adaptation_next_iter}
    \vec{a}^{l+1} = \vec{a}^l + g \cdot \Delta \vec{a}^{l+1}
  \end{equation}
  строится итерационная вычислительная процедура.

%l: В алгоритме можно распараллелить вычисл-е производных -> 
% моделир-ть несколько СГ независимо; + СЛАУ
% + в каждой СГ параллельные вычисления по полной программи
% Параллельные + распределённые вычисления
  В данном алгоритме технологии параллельных и распределённых вычислений 
  могут быть применены при расчете $n_a$ частных производных 
    $\frac {\partial p_r^l} {\partial a_k^l}$ 
  как независимых друг от друга вычислительных процессов, а также 
  при решении системы линейных уравнений \eqref{eq:adaptation_chuvst_final}.

%l: Расчёт n_a + 1 режимов СГ при различных \vec{a} на каждой итерации пар-но.
  Для численного расчета частных производных параметров режима 
  в контрольных точках СГ от идентифицируемых параметров модели 
  необходимо рассчитать $n_a + 1$ режимов СГ 
  при различных состояниях вектора идентифицируемых параметров $\vec{a}$ модели. 
  Таким образом, можно организовать параллельный вычислительный процесс 
  расчёта режимов СГ на каждой итерации. 

%l: Размерность задачи ещё многократно возрастёт, если временные ряды замеров.
  Размерность задачи может многократно возрасти, если использовать 
  множество замеров не для одного момента времени, 
  а на различных уровнях квантования рассматриваемого отрезка времени 
  $t = [t_0, t_j]$, 
  то есть каждый контролируемый параметр режима 
  будет представлять собой временной ряд замеров.

%l: Целесообразность распределения нагрузки в распр-й среде
    Задача расчёта одного экземпляра режима СГ может полностью использовать 
    вычислительные ресурсы отдельного компьютера, 
    поэтому параллельные процессы расчёта множества режимов СГ 
    целесообразно организовать в распределённой среде, 
    что можно сделать следующим образом.
    \begin{enumerate}
    \item % 1. Запуск сервиса идентификации на КВР
      Вычислительный сервис процедуры адаптации запускается на комплекте 
      вычислительных ресурсов, приспособленном для расчёта режима СГ.
    \item % 2. Запрос и получение доп. КВР сервисом идентификации
      Сервис адаптации запрашивает у сервиса-диспетчера вычислительных ресурсов
      дополнительные комплекты вычислительных ресурсов и получает их.
    \item % 3. Запуск сервисов моделирования СГ на звахваченных КВР.
      Сервис адаптации запускает на каждом из захваченных ресурсов сервис расчёта
      режима СГ и вызывает метод загрузки адаптируемой СГ на каждом из них.
    \item % 4. Очередь всех векторов a
      Сервис адаптации на каждой итерации метода чувствительности формирует 
      очередь из всех значений идентифицируемых параметров модели $\vec{a}$, 
      которые необходимо рассчитать для вычисления производных адаптируемых 
      параметров режима $\vec{p}$ по идентифицируемым параметрам модели $\vec{a}$.
    \item % 5. Асинхронный запуск расчётов на КВР до исчерпания очереди
      Сервис адаптации в цикле до исчерпания очереди извлекает значения вектора 
      идентифицируемых параметров модели $\vec{a}$, 
      асинхронно передаёт эти значения на свободный вычислительный ресурс и 
      запускает вычислительный процесс расчёта режима. 
      Как только вычислительный ресурс заканчивает расчёт СГ, 
      то есть сформирован вектор адаптируемых параметров режима $\vec{p}$, 
      сервис адаптации получает очередное задание.
    \item % 6. Сбор данных, решение СЛАУ, закончить, или след. итерация
      После расчета всех производных сервис адаптации формирует 
      систему уравнений~\eqref{eq:adaptation_min_condition_SLAE}, 
      решает её с помощью сервиса решения систем уравнений, 
      формирует новое состояние вектора идентифицируемых параметров модели 
      $\vec{a}$ и если критерий завершения процедуры адаптации не выполняется, 
      переходит к организации выполнения вычислительного процесса 
      следующей итерации.
    \end{enumerate}

%l: Перечисление сервисов, разработанных на основе архитектуры
На основе предложенной архитектуры разработан ряд вычислительных сервисов. 
\begin{itemize}
  \item % СЛАУ
    Взаимозаменяемое семейство решения систем уравнений: 
    на базе Intel MKL для плотных матриц, Intel MKL для разреженных матриц, 
    CuSp для разреженных матриц на CUDA.
  \item % Обобщённые и оптимизированный менеджер параллельного моделирования
    Обобщённые менеджер параллельного моделирования расчётных объектов 
      на центральном процессоре с использованием OpenMP, Intel TBB; 
      на CUDA-устройствах с использованием thrust; 
    оптимизированный менеджер параллельного моделирования трубопроводов 
    на CUDA C с использованием эффективного для работы с GPU 
    расположения данных в памяти.
  \item % Севрис моделирования СГ
    Сервис моделирования режимов СГ.
  \item % Сервис адаптации СГ (не весь вообще-то, а только основная часть)
    Сервис расчёта множества режимов СГ при различных значениях 
    идентифицируемых параметрах расчетной модели режима.
  \item % Управляющие сервисы: УС КВР, диспетчер
    Управляющие сервисы комплектами ресурсов и диспетчер ресурсов.
\end{itemize}

%l: Анализ и выводы по вычислительным экспериментам
  Анализ вычислительных экспериментов позволил сделать следующие выводы.
  \begin{itemize}
    \item % Гетерогенные ПВ, ускорение в рамках одного ПК
      Предложенные архитектурные решения позволяют значительно ускорить 
      параллельное моделирование векторов расчётных объектов за счёт 
      использования гетерогенных параллельных вычислений~--- 
      получено ускорение до 30 раз в рамках одного компьютера.
    \item % Разреженные решатели перспективны - стоит исследовать
      Целесообразно более детально исследовать 
      разреженные решатели систем уравнений, 
      так как на тестовых примерах они многократно опережали 
      решатели для плотных систем по скорости и 
      экономичности потребления памяти. 
    \item % Сложные задачи масштабируются в распределённой среде
      Вычислительно-сложные задачи моделирования, 
      основанные на моделировании множества режимов СГ, 
      хорошо масштабируются в распределённой среде~--– 
      получено ускорение в $3.67$ раз 
      при распределении нагрузки между 4-мя узлами, 
      соединёнными локальной сетью при моделировании режимов СГ 
      с различными значениями параметров адаптации.
    \item % Целесообразно использование облаков, сильно-удалённых ресурсов
      Целесообразно решение вычислительно сложных задач на удалённых 
      вычислительных ресурсах~--– несмотря на существенные накладные расходы 
      (порядка 8\% от общего времени выполнения операции) 
      достигнуто 10-кратное ускорение расчёта режимов СГ 
      в облаке AWS (Amazon Web Services) 
      на группе из четырёх виртуальных машин 
      по сравнению с локальным расчётом.
  \end{itemize}
